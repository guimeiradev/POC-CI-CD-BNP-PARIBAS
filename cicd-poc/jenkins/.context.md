Build context for the custom Jenkins image used by the `jenkins` service in
`cicd-poc/docker-compose.yml`. Extends the official Jenkins LTS image with the
.NET SDK and the tooling the pipeline and (planned) quality gates need.

## Summary

- Dockerfile — based on `jenkins/jenkins:lts-jdk17`; installs `curl`, `libicu-dev`, `zip`; installs the .NET 10 SDK via Microsoft's `dotnet-install.sh`; installs the `dotnet-sonarscanner` and `dotnet-reportgenerator-globaltool` global tools; installs Semgrep into a Python venv (`/opt/semgrep-venv`) to work around PEP 668 on Debian Bookworm; installs Jenkins plugins from `plugins.txt`.
- plugins.txt — Jenkins plugin list installed via `jenkins-plugin-cli`: `workflow-aggregator`, `git`, `github`, `sonar`, `ansible`, `credentials-binding`, `pipeline-stage-view`, `blueocean`, `ws-cleanup`, `timestamper`, `build-timeout`, `dependency-check-jenkins-plugin` (all pinned to `:latest`).

## Constraints

- MUST install the .NET SDK via `dotnet-install.sh` (not `apt`) to control the exact installed version (per root `README.md` design decisions).

## Guidance

- Semgrep, the SonarQube scanner tooling, and the `dependency-check-jenkins-plugin` are installed in this image but, as of this scan, no stage in the root `Jenkinsfile` invokes them yet — the pipeline only runs Restore/Build/Test/Publish/Archive/Upload to Nexus. Wiring them into pipeline stages is tracked as the planned Fase 3 (Quality Gates) work per the root `README.md` roadmap.
- Plugin versions are pinned to `:latest` in `plugins.txt`, which SHOULD be reviewed if build reproducibility becomes a concern.
