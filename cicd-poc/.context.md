Docker Compose definition for the local CI/CD toolchain (Jenkins, SonarQube,
Postgres, Nexus, Vault) plus the manual setup steps required before the
Jenkins pipeline in the repo root can run end to end.

## Summary

- docker-compose.yml — defines the `cicd-net` bridge network and five services: `jenkins` (custom-built image), `sonarqube` (community edition), `db` (Postgres 16, SonarQube's backing store), `nexus` (Sonatype Nexus3, artifact storage), `vault` (HashiCorp Vault in dev mode).
- README.md — prerequisites (Docker, `vm.max_map_count` for SonarQube's Elasticsearch), build/start commands, service URLs and credentials, and the Fase 2 manual setup checklist (Jenkins rebuild, Nexus repo creation, Jenkins credentials, Pipeline job creation).
- jenkins/ — the custom Jenkins image build context.

## Constraints

- The `sonarqube` service MUST use the `db` (Postgres) service as its JDBC backend — SonarQube does not support its embedded H2 database in this setup (per `README.md` design decisions).
- The `nexus` repository used by the pipeline MUST be named `dotnet-artifacts` with write policy `ALLOW` (redeploy) — the `Jenkinsfile` and README setup steps assume this exact name and policy.

## Guidance

- SHOULD run `sudo sysctl -w vm.max_map_count=262144` before `docker compose up` — SonarQube's Elasticsearch will fail to start otherwise.
- Credentials in `docker-compose.yml` are hardcoded intentionally for local PoC use only (per README "Decisões de design"); this SHOULD NOT be treated as a production pattern.
